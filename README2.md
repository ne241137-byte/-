# ドラレコ録画システム 処理ロジック説明

この資料は、Rustで作成したドラレコ録画システムについて、「何がトリガーになって、何が起きるか」を説明するものです。

## 1. システム全体の目的

このプログラムは、Raspberry Pi 5またはMac上で、フロントカメラとリアカメラの2台を同時に動かし、常時録画を行うドラレコ用の録画システムです。

録画データは、スマートフォン向けのローカルWebビューアで見やすいように、`videos/` フォルダ配下へ整理して保存します。
インターネットやネットワーク接続は常時使う前提ではなく、録画を確認したい時だけローカル接続または転送モードを使う想定です。

## 2. 保存フォルダ構成

録画データは次のように保存されます。

```text
videos/
  front/
    continuous/  フロント常時録画
    event/       フロント事故イベント録画
    manual/      フロント手動録画
  rear/
    continuous/  リア常時録画
    event/       リア事故イベント録画
    manual/      リア手動録画
```

`continuous` は通常の常時録画です。

`event` は事故や衝撃などを想定したイベント保存用です。

`manual` は運転者が手動で保存したいときの保存用です。

## 3. 起動時に起きること

プログラムを起動すると、最初に以下の処理が行われます。

1. `ffmpeg` が使えるか確認する
2. `videos/` フォルダを作成する
3. フロントカメラの設定を作る
4. リアカメラの設定を作る
5. フロント録画用スレッドを起動する
6. リア録画用スレッドを起動する
7. キー入力を受け付ける状態になる

フロントとリアは別々のスレッドで動きます。そのため、2つのカメラは同時に常時録画されます。

## 4. カメラ設定

Macではカメラ入力方式として `avfoundation` を使います。

Raspberry Pi / Linuxでは `v4l2` を使います。

初期設定は以下です。

```text
解像度: 1280x720
FPS: 30
動画形式: mp4
映像コーデック: H.264
映像ビットレート: 3Mbps
ラップ長: 10分
```

Raspberry Pi 5では、通常は以下のように使います。

```text
フロントカメラ: /dev/video0
リアカメラ: /dev/video1
```

ただし、接続環境によって `/dev/video2` などになることがあります。その場合は実行時に指定します。

## 5. 常時録画の流れ

常時録画は、プログラム起動後に自動で始まります。

各カメラでは以下の処理を繰り返します。

1. カメラを初期化する
2. 現在時刻からファイル名を作る
3. 10分間の録画を開始する
4. 録画が終わったら `.mp4` として保存する
5. サムネイル画像 `.jpg` を作成する
6. 必要に応じてイベント保存または手動保存を処理する
7. 容量が上限を超えていれば古い常時録画を削除する
8. 次の10分録画へ進む

常時録画ファイル名の例:

```text
front_video_20260714_120000.mp4
rear_video_20260714_120000.mp4
```

サムネイル名の例:

```text
front_thumb_20260714_120000.jpg
rear_thumb_20260714_120000.jpg
```

## 6. aキーの処理 事故イベント保存

### トリガー

```text
a
```

または

```text
ACCIDENT
```

### 起きること

事故イベント保存が予約されます。

この時点ですぐに動画をコピーするのではなく、「次のラップが終わるまで待ってから」保存します。

理由は、要件上、以下の3つを保存するためです。

```text
前の1ラップ
現在録画中の1ラップ
次の1ラップ
```

つまり、事故イベントではフロントとリアそれぞれで3本ずつ、合計6本の動画を保存します。

保存先の例:

```text
videos/front/event/ACCIDENT_20260714_120000/
videos/rear/event/ACCIDENT_20260714_120000/
```

保存されるファイル名には `EVENT_` が付きます。

## 7. eキーの処理 手動保存

### トリガー

```text
e
```

または

```text
EVENT
MANUAL
```

### 起きること

手動保存が予約されます。

事故イベントと同じように、以下の3つのラップを保存します。

```text
前の1ラップ
現在録画中の1ラップ
次の1ラップ
```

フロントとリアそれぞれで3本ずつ、合計6本の動画を保存します。

保存先の例:

```text
videos/front/manual/MANUAL_20260714_120000/
videos/rear/manual/MANUAL_20260714_120000/
```

保存されるファイル名には `MANUAL_` が付きます。

## 8. iキーの処理 カメラ初期化

### トリガー

```text
i
```

または

```text
INIT
```

### 起きること

フロントカメラとリアカメラの両方に、初期化リクエストを出します。

録画中の10分ラップを途中で壊さないため、即座に強制停止するのではなく、次のラップ開始前に初期化を行います。

初期化では、`ffmpeg` を使ってカメラから1フレーム取得できるかを確認します。

## 9. fキーの処理 録画停止

### トリガー

```text
f
```

または

```text
STOP
```

### 起きること

録画停止フラグを立てます。

フロント録画スレッドとリア録画スレッドは、その停止フラグを見て録画を終了します。

録画中の `ffmpeg` プロセスも停止されます。

最後に、プログラム全体が終了します。

## 10. bキーの処理 バック時のフロントカメラ一時停止

### トリガー

```text
b
```

または

```text
BACK
```

### 起きること

フロントカメラだけを一時停止します。

リアカメラはそのまま録画を続けます。

これは、車がバックしている間はリア側の確認を優先し、フロント側の録画処理を一時的に止める運用を想定したものです。

もう一度 `b` を押すと、フロントカメラ録画を再開します。

現在録画中のフロント動画がある場合は、その録画プロセスを停止し、途中で止まったファイルは通常録画として扱わないように削除します。

## 11. 容量管理の考え方

常時録画は放置すると容量を使い続けます。

そのため、各カメラごとに最大容量を持っています。

初期値は以下です。

```text
フロント: 30GB
リア: 30GB
合計: 約60GB
```

常時録画フォルダの容量が上限を超えた場合、古い動画から削除します。

ただし、事故イベントや手動保存として保護対象になっている動画は削除しません。

## 12. 環境変数で変更できる設定

実行時に環境変数を指定することで、カメラや録画設定を変更できます。

```text
FRONT_DEVICE                  フロントカメラ
REAR_DEVICE                   リアカメラ
DASHCAM_FPS                   FPS
DASHCAM_SIZE                  解像度
DASHCAM_VIDEO_CODEC           動画コーデック
DASHCAM_VIDEO_BITRATE         動画ビットレート
DASHCAM_MAX_BYTES_PER_CAMERA  カメラ1台あたりの最大保存容量
DASHCAM_MEDIA_DIR             保存先フォルダ
DASHCAM_INPUT_FORMAT          入力方式
```

Raspberry Pi 5での実行例:

```sh
FRONT_DEVICE=/dev/video0 REAR_DEVICE=/dev/video1 DASHCAM_FPS=30 DASHCAM_SIZE=1280x720 cargo run --offline
```

もしリアカメラが `/dev/video2` の場合:

```sh
FRONT_DEVICE=/dev/video0 REAR_DEVICE=/dev/video2 DASHCAM_FPS=30 DASHCAM_SIZE=1280x720 cargo run --offline
```

## 13. このコードで実装している範囲

このコードで実装しているのは、ドラレコ録画基盤です。

実装済み:

```text
フロント/リア2カメラ録画
10分ラップ録画
mp4保存
H.264 / 3Mbps指定
常時録画
事故イベント保存
手動保存
前1ラップ・現在1ラップ・後1ラップ保存
容量上限による古い録画の削除
スマホWebビューア向けフォルダ構成
バック時のフロントカメラ一時停止
```

未実装または別プロセス想定:

```text
YOLO/OpenCVによる画像認識
TTC衝突予知
IMU/GPIOによる自動事故検知
音声AACの同時録音
レンズ汚れ診断
録画確認時だけWebビューアへ切り替える制御
```

## 14. 簡単な処理図

```text
プログラム起動
  ↓
ffmpeg確認
  ↓
videosフォルダ作成
  ↓
フロント録画スレッド起動
リア録画スレッド起動
  ↓
10分ごとにmp4保存
  ↓
キー入力待ち
  ├─ a → 事故イベント保存予約
  ├─ e → 手動保存予約
  ├─ i → カメラ初期化予約
  ├─ b → フロントカメラ一時停止/再開
  └─ f → 録画停止
```

## 15. 開発用キーと実際のドラレコ運用での対応

現在のプログラムでは、開発・検証しやすいようにキーボード入力で操作しています。

実際に車載ドラレコとして運用する場合は、これらのキー入力はディスプレイ上の操作、センサー、車両電源、親システムからの制御信号に置き換わります。

| 現在のキー | 現在の機能 | 実際のドラレコでの相当機能 | 実際のきっかけ |
|---|---|---|---|
| `a` | 事故イベント保存 | イベント記録 | 衝撃センサー、IMU、急ハンドル検知、急ブレーキ検知など |
| `e` | 手動保存 | 手動記録 | 運転者がディスプレイ上の保存ボタンを押す |
| `i` | カメラ初期化 | カメラ再初期化・自己復旧 | 起動時、カメラ異常検知時、メンテナンス操作時 |
| `b` | フロントカメラ一時停止/再開 | バック時のフロント録画抑制 | シフトがRに入る、バック画面表示、親システムからのバック信号 |
| `f` | 録画停止 | システム停止 | 車両電源OFF、親システムからの停止指示、メンテナンスモード移行 |
| プログラム起動 | 常時録画開始 | イグニッションON連動の録画開始 | 車両電源ON、Raspberry Pi起動、systemdサービス起動 |

### aキーの実運用上の意味

`a` キーは、現在は事故イベント保存を手動で試すための開発用トリガーです。

実際の車載運用では、以下のようなセンサー検知に置き換わります。

```text
一定以上の衝撃Gを検知
急ハンドルを検知
急ブレーキを検知
車両挙動の異常を検知
ADAS側が衝突危険を確定
```

これらが起きたとき、プログラム内部では現在の `a` と同じ処理を呼び出します。

その結果、フロントとリアの両方で以下を保存します。

```text
前の10分ラップ
現在録画中の10分ラップ
次の10分ラップ
```

### eキーの実運用上の意味

`e` キーは、運転者が「この映像を残したい」と判断したときの手動保存に相当します。

実際の車載運用では、車内の物理ボタンは使わず、ディスプレイ上の保存ボタンに置き換わります。

```text
車載ディスプレイ上の保存ボタン
スマートフォン画面上の保存ボタン
ローカルWebビューア上の保存ボタン
親システムからの手動保存コマンド
```

保存される範囲は事故イベントと同じく、前1ラップ、現在1ラップ、後1ラップです。

### iキーの実運用上の意味

`i` キーは、カメラの再初期化に相当します。

実際の車載運用では、通常は運転者が頻繁に押すものではなく、以下のような場面で使われます。

```text
システム起動直後
カメラが一時的に認識できなくなったとき
USBカメラの再接続後
映像入力異常を検知したとき
整備・検証時
```

現在のコードでは、録画中の10分ファイルを壊さないため、`i` が押されても即座に録画を中断せず、次のラップ開始前に再初期化します。

### bキーの実運用上の意味

`b` キーは、バック時にフロントカメラを一時停止するための開発用トリガーです。

実際の車載運用では、以下のようなきっかけに置き換わります。

```text
シフトがRに入る
バック画面に切り替わる
親システムからバック信号を受け取る
ディスプレイ上でバックモードが有効になる
```

この状態では、フロントカメラだけ録画を止め、リアカメラは録画を続けます。

バックが終わったら、もう一度 `b` 相当の信号でフロント録画を再開します。

### fキーの実運用上の意味

`f` キーは、録画停止に相当します。

実際の車載運用では、以下のようなきっかけに置き換わります。

```text
車両電源OFF
Raspberry Piのシャットダウン処理
親システムからの停止命令
録画確認用のWeb転送モードへ移行する前の安全停止
メンテナンスモード開始
```

要件定義書では、Web転送を行う際には録画やADAS推論を安全停止してから転送モードに入る想定です。
つまり、常にインターネットへ接続して動画を送るのではなく、ユーザーが録画を確認したい時だけ接続する運用です。

そのため、将来的には `f` 相当の停止処理を親システムが呼び出し、その後にローカルWebビューアや転送サーバーを必要な時だけ起動する流れになります。

## 16. 実運用時のトリガー置き換えイメージ

現在:

```text
キーボード入力
  a / e / i / b / f
```

実運用:

```text
車両電源ON
  → プログラム起動
  → 常時録画開始

IMUが衝撃を検知
  → 事故イベント保存

運転者がディスプレイ上の保存ボタンを押す
  → 手動保存

カメラ異常を検知
  → カメラ初期化

シフトがRに入る
  → フロントカメラ一時停止、リア録画継続

バックが終わる
  → フロントカメラ再開

車両電源OFFまたは録画確認用の転送モード開始
  → 録画停止
```

つまり、現在のキー操作は最終製品に残る操作そのものではなく、ディスプレイ操作や実機センサーをつなぐ前に、各機能を確認するための仮トリガーです。


# 16. 開発用キーと実運用時の対応

| 開発用キー | 現在の機能 | 実運用でのトリガー（きっかけ） |
|------------|-----------|--------------------------------|
| **aキー** | 事故イベント保存 | 衝撃センサー（Gセンサー）、IMU、急ハンドル検知、急ブレーキ検知、衝突予測（ADAS）など |
| **eキー** | 手動保存 | 車内の物理保存ボタン、GPIO接続の保存ボタン、車載ディスプレイの保存ボタン、スマホアプリ・Web画面の保存ボタン |
| **iキー** | カメラ初期化 | システム起動時、カメラ異常検知時、USBカメラ再接続時、映像取得失敗時、整備・メンテナンス時 |
| **bキー** | フロントカメラ一時停止／再開 | シフトレバーがRに入る、バックカメラ表示開始、親システムからバック信号受信 |
| **fキー** | 録画停止 | 車両電源OFF、録画データ転送モードへの移行、親システムからの停止指示、シャットダウン開始 |
| **プログラム起動** | 常時録画開始 | イグニッションON、車両電源ON、Raspberry Pi起動、systemdサービス起動 |

## 各キーの実運用上の意味

- **aキー**：開発中はキーボード入力で事故イベント保存を確認します。実運用では衝撃センサー、IMU、急ハンドル・急ブレーキ検知、ADASなどが同じ処理を自動実行します。
- **eキー**：開発中はキーボード入力ですが、実運用では物理ボタン、GPIOボタン、車載ディスプレイ、スマホ・Web画面の保存ボタンから実行されます。
- **iキー**：開発中は手動初期化ですが、実運用では起動時、カメラ異常、USB再接続、映像取得失敗、整備時などに実行されます。
- **fキー**：開発中は手動停止ですが、実運用では車両電源OFF、転送モード移行、親システムからの停止指示、シャットダウン開始時に実行されます。
